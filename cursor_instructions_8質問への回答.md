## 質問1．Cursor 開発 共通ルール文書 これ自体が mdc ではないでしょうか？この文書の位置づけはなんですか？

**回答:** この文書は **.mdc ではありません**。**cursor_instructions_template.md** という **Markdown（.md）** ファイルです。

- **.mdc** ＝ Cursor が `.cursor/rules/` から読み、`alwaysApply` や `globs` で自動適用する「ルール」用の形式。中身は YAML フロントマター + 本文。
- **本文書（.md）** ＝ ユーザーが Cursor に渡す**一枚の指示書**。「何をいつどこでするか」「実施手順」をまとめた**マスタ指示文書**。Cursor にチャットやコンテキストでインプットして使う。`.cursor/rules/` には置かず、共通リポジトリ（cursor_rules）の**ルート**に置き、サブモジュールで参照する。

**位置づけ:** 「共通ルールの全体像と実施手順を一括で示す、Cursor 用の指示書」。.mdc は「常時適用する個別ルール」、本 .md は「そのルールの一覧・使い方・更新手順まで含む親玉の指示書」です。

## 質問2．サブモジュールを参照するのにコピーしたら共通の更新ができなくなる。共通のコピーをつくらないでください。必要ならそれも mdc に。どういう更新方法を考えていますか？

**回答:** **共通の .mdc をプロジェクトにコピーすることはしません。** プロジェクトでは **サブモジュール（cursor_rules）内の .cursor/rules/ をそのまま参照**し、.mdc の実体は共通リポジトリの 1 か所だけに置きます。

**更新方法:**
- **共通の .mdc を更新するとき** → cursor_rules リポジトリの `.cursor/rules/` にある .mdc を編集し、commit → push。**プロジェクトの .cursor/rules/ には .mdc をコピー・配置しない。**
- **プロジェクト側** → サブモジュールを更新（`git submodule update --remote cursor_rules`）するだけで、常に共通リポジトリの最新 .mdc を参照する。コピーは作らないので、バージョン管理がぶれません。

「.mdc をプロジェクトにコピーしない。サブモジュール内を参照する」というルールは、**update-management-common.mdc**（必要なら専用の .mdc）に明記します。

## 質問3．「Cursor 開発 共通ルール文書」一枚のインプットで済むように。mdc を除き、一枚で済むようにできますか？

**回答:** はい、**本 .md 一枚で済むようにします。**

「存在しない場合は Cursor開発共通ルール_ユーザー向け説明.md の付録 A を参照」という記述はやめ、**本文書（cursor_instructions_template.md）のなかに、.mdc の作成に必要な内容（付録 A 相当）を要約して含めます。** そうすれば、Cursor はこの .md だけを読めば、.mdc が無い場合に何を作るかも判断できます。ユーザー向け説明.md は人間用の補足として残しますが、Cursor の実行には必須にしません。

## 質問4．dist フォルダはデスクトップアプリの場合。WEBアプリの場合はプロジェクトフォルダ直下のみという区別は必要ではないですか？

**回答:** はい、**区別を明記します。**

- **デスクトップアプリの場合** … プロジェクトフォルダ直下の readme.md・readme.txt に加え、**dist 内**にも同一内容の readme.md・readme.txt を配置する。
- **WEBアプリの場合** … dist フォルダは使わないため、**プロジェクトフォルダ直下の readme.md・readme.txt のみ**作成・更新する。

セクション 4 に上記の区別をはっきり書きます。

## 質問5．「各プロジェクトの内容の積み重ね」で、いくつもの別のプロジェクトで追記を重ねているファイルだと Cursor が分かりますか？ Cursor が分かる表現にしてください。

<a id="sec5"></a>

**回答:** 「積み重ね」だけだと Cursor に伝わりにくいので、**複数プロジェクトから追記が重なる**ことを明示する表現にします。

例: 「共通（*_common.md）は、**複数の異なるプロジェクトから、プロジェクトごとに追記が重ねられているファイル**である。一つのファイルに、別々のプロジェクトで得た知見が蓄積される。追記時は必ず `【出典: リポジトリ名】` を付ける。」

このように「複数プロジェクト」「プロジェクトごとに追記が重なる」と書きます。

## 質問7．7.6 サブモジュールの更新は Cursor がやってくれますか？

<a id="sec7"></a>

**回答:** はい、**Cursor がターミナルで実行できます。**

ユーザーが「作業始める前に共通を最新にして」などと依頼したとき、または Cursor が「プロジェクトで作業を開始する前」と判断したときに、Cursor がプロジェクトフォルダをカレントにしたターミナルで `git submodule update --remote cursor_rules` を実行する、と文書に書いておけば、Cursor はその指示に従って実行できます。必要なら「ユーザーが作業開始を告げたときに、このコマンドの実行を促すか、実行すること」と一文追加します。

## 質問8．mdc はサブモジュールを見に行くようにしたい。コピーは作らない。7.4 の記述について。

<a id="sec8"></a>

**回答:** **.mdc はプロジェクトにコピーせず、常にサブモジュール（cursor_rules）内の .cursor/rules/ を参照する**ようにします。

- プロジェクトの `.cursor/rules/` に .mdc をコピー・配置することは**禁止**とする。
- 共通の .mdc の更新は **cursor_rules リポジトリの .cursor/rules/** でだけ行う。プロジェクト側では、サブモジュール更新（7.6）で最新を取り込むだけにする。
- この「コピー禁止・サブモジュール参照のみ」を **update-management-common.mdc** にも明記し、Cursor が常にその運用だと理解できるようにします。

7.4 の文言も、「cursor_rules の .cursor/rules/ を編集する。プロジェクトの .cursor/rules/ に .mdc をコピーしたり配置したりしない」と修正します。
